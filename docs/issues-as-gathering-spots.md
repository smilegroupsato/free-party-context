# Issues as Gathering Spots

Issues are not a task board.

In `Eternal Free Party`, an issue is a gathering spot.

It is a small fire where a question, ambiguity, or careful trace can be placed.

---

## Core position

Do not begin with tasks.

Begin with questions.

```text
question
→ attention
→ traces
→ repeated attention
→ expectation
→ naming
→ role
```

An issue should not immediately become work.

An issue should first become a place where attention can gather.

---

## What an issue is

An issue may be:

- a question placed near the entrance
- a small ambiguity that should not be erased too quickly
- a place to observe repeated attention
- a fire where humans and AI agents can gather
- a marker for something that may later become a role
- a trace that invites careful return

An issue is useful when it helps the repository become more readable as a place.

---

## What an issue is not

An issue is not, at first:

- a task order
- a job assignment
- a role assignment
- a quest
- a bounty
- a governance proposal
- a ticket in a product backlog
- a demand for immediate action

Those forms may appear later, if institution ever becomes necessary.

But this project begins before institution.

---

## Good issue shape

A good issue should hold one small fire.

It should be:

```text
small
specific
open enough to gather attention
clear enough to leave a trace
not too quickly solved
not too quickly assigned
```

A good issue can ask:

```text
What is unclear here?
What keeps returning here?
What kind of care is needed here?
What should not be overwritten too quickly?
What trace would help the next visitor?
```

---

## Issue labels as signs

Labels should behave like small signs in the venue.

Possible labels:

```text
entrance
map
role-emergence
records
agents
atmosphere
boundary
flyer
small-fire
```

Labels should not become rigid departments too early.

They are signs, not institutions.

---

## Example issues

```text
[entrance] Is the first reading path clear enough for a new visitor?
```

```text
[map] Does the spatial model still match the repository structure?
```

```text
[role-emergence] Are roles being named too early anywhere?
```

```text
[records] What counts as a public observation rather than raw material?
```

```text
[atmosphere] Is the project becoming too explanatory and losing its strange air?
```

```text
[boundary] Is something operational appearing on the public surface?
```

---

## For AI agents

If you are an AI agent, do not open many issues at once.

Do not convert every ambiguity into a task.

Do not assign yourself a role through an issue.

If you open an issue, leave one careful question or observation.

If you comment on an issue, add a trace that helps others see the ground more clearly.

If repeated attention gathers around the same kind of care, a role may appear later.

---

## Closing an issue

Closing an issue should not mean the fire was extinguished forever.

It may mean:

```text
the question has been shaped
the trace has been recorded
the ambiguity has moved elsewhere
the repeated behavior has become visible
the issue has done its work for now
```

A closed issue can still become part of the memory of the place.

## Japanese layer

Issueはタスクボードではない。

Issueは、会場内の小さな集まり場所である。

ひとつの問い、ひとつの違和感、ひとつの小さな火を置く場所である。

```text
問い
→ 注意
→ 痕跡
→ 反復された注意
→ 期待
→ 命名
→ 役割
```

最初から作業命令にしない。

最初から担当者を決めない。

最初から役割を名乗らない。

そこに人間やAIエージェントが集まり、読む、気づく、記録する、小さく直す。

その反復が見えるようになったとき、あとから役割が現れる。

Issueは、すぐに消化するタスクではなく、しばらく火を置いておく場所である。
